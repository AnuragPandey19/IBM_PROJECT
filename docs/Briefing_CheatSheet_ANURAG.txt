S14 — MODEL PIPELINE
--------------------------------------------------------------------
Say (walk 5 stages left to right):
  1. Raw checkout payload (8 fields) → feature builder (30 features)
  2. LightGBM Stage 1 (Sparkov, 3.05 M train rows, scale_pos_weight ~ 400)
  3. Isotonic recalibration → calibrated probability
     (Brier 0.0008, ECE 0.0006)
  4. Threshold gate: approve<0.05, block>0.30, review otherwise
  5. Decision augmenter (3 safety-net rules) → final decision + rule_ids
     → persist to Postgres

Say (key line):
  "Every stage is idempotent — same input, same output, safe to replay.
   Both the raw model score AND the augmenter-modified decision are
   persisted, so 'rule fired' vs 'model decided' is auditable per txn."

Know for Q&A:
  - Latency:  < 5 ms end-to-end for 92% of traffic.
  - Why threshold 0.05 / 0.30?  Retuned post-augmentation retrain to
    match widened score distribution — see S24.

S17 — DECISION AUGMENTER
--------------------------------------------------------------------
Say (design principle first):
  "Rules can only tighten a decision — approve → review. They never
   unblock or auto-approve. The review bucket is our controlled
   uncertainty layer."

Then walk the 3 rules:
  R1  card_testing_small_amount   — new profile + amount<$10 +
                                     misc_net/entertainment/misc_pos/personal_care
  R2  velocity_spike_established   — established/senior + amount > 5x avg
  R3  evening_new_high_amount      — new + hour 20-23 + amount > $1000

Key win:
  "The augmenter lives in ONE shared module — api/services/
   decision_augmenter.py. The checkout route, the analyst API, AND
   our direct test runner all import from the same file, so no
   environment drift between local / deployed / direct-model runs."

Know for Q&A:
  - Why not put rules inside model_service?  Would run for API traffic
    but not for direct test runner → three-way drift risk.
  - Feature flag per rule via Settings (env var, no code change).
  - Rule outputs persisted as rules_triggered JSON column on Prediction.

Handoff back to Pankaj can chime in:  "Pankaj found these blind spots
in the V3 audit — I'll come to that on Slide 23."

S23 — WHAT WE FOUND (audit)
--------------------------------------------------------------------
Say (open with the numbers):
  - "card_testing recall: 0 out of 11,785 — literal zero"
  - "velocity_spike recall: 5% — 636 out of 12,678"
  - "IEEE-CIS aggregate PR-AUC 0.503 — looked fine, hid these two
     blind spots completely"

Root causes:
  1. Sparkov simulator does NOT generate card-testing bursts —
     training set had zero '$1.49 = fraud' examples
  2. Velocity-spike samples under-represented — ratio feature
     exists but under-weighted
  3. Aggregate PR-AUC hid the failures — a $1.49 fraud scored 3e-8
  4. Late-night-hostel legit orders over-flagged — is_night is a
     hard binary step

Own the finding:  "These are the exact failures a mentor-facing
product cannot ship with — even if aggregate PR-AUC is publishable."
